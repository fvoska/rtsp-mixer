### `ReconnectSupervisor`

Constructor: three required callbacks + optional Random.

```dart
ReconnectSupervisor({
  required Future<void> Function(String cameraId) onAttempt,
  required void Function(String cameraId, ReconnectStatus status) onStatusChange,
  required void Function(ReconnectEventType type, String cameraId, String? detail) onEvent,
  Random? random,
});
```

Public methods:
- `Future<void> requestReconnect(String cameraId, {required String cause, bool immediate = false})` â€” entry point for all triggers (stream.error, stream.completed today; zombie in 04-02; wifi in 04-04). Dedup via `inFlight || hasPendingRetry`. `immediate=true` bypasses the backoff delay (for WiFi-reconnect).
- `void cancelAll()` â€” cancels every `retryTimer` + `alertTimer` and clears per-camera state. Idempotent.
- `int attemptCount(String cameraId)` and `bool hasPendingRetry(String cameraId)` â€” test hooks, but also usable by future UI (e.g., "retrying in ~8sâ€¦" countdown).

### Callback Contracts (caller's obligations)

- **onAttempt(cameraId)**: MUST stop + reopen the player. MAY throw; supervisor treats that as a failed attempt and schedules the next backoff-delayed retry. Should not block beyond ~15s (wrap open() in a timeout).
- **onStatusChange(cameraId, status)**: called twice per successful reconnect â€” `reconnecting` on kickoff, `playing` on success. Caller maps to `CameraConnectionStatus` via `copyWithCamera`.
- **onEvent(type, cameraId, detail)**: called twice per reconnect â€” `reconnectAttempt` (with `'attempt N (cause=X)'` detail string) + `reconnectSuccess` on recovery. Caller forwards to `healthEventsProvider.notifier.record()`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Dedup guard widened to cover pending-retry window**
- **Found during:** Task 2
- **Issue:** The plan's `Pattern 3` implementation inside `requestReconnect` uses `try { â€¦ } finally { st.inFlight = false; }` so `inFlight` flips back to `false` before the scheduled retry Timer ever fires. The plan's own `trigger_dedupe_test.dart` test ("two simultaneous requests for the same camera produce one attempt") would fail because the second call enters with `inFlight == false` and schedules its own parallel Timer.
- **Fix:** Check `inFlight || (retryTimer?.isActive ?? false)` on entry. While a retry is scheduled, new triggers for the same cameraId are suppressed â€” which is what the test expects and what the dedup semantic should be.
- **Files modified:** `lib/features/monitoring/services/reconnect_supervisor.dart`
- **Commit:** `2942ecd`

**2. [Rule 1 - Bug] Avoided double-scheduling after failed attempt**
- **Found during:** Task 2
- **Issue:** The plan pattern has `_attemptReconnect` reschedule on failure AND `_scheduleRetry`'s outer catch also reschedule (`st.attempt += 1; _scheduleRetry(...)`). The catch branch would execute because `_attemptReconnect` rethrows. This means each failure spawns two parallel retries â€” exactly the bug the dedup is supposed to prevent, now triggered from inside the supervisor itself.
- **Fix:** Kept the rescheduling in `_attemptReconnect`'s own catch block (single source of truth), left the Timer's outer catch as pure logging. The three-layer defensive pattern is preserved via (a) `_attemptReconnect` inner try/catch â†’ schedule next, (b) `_scheduleRetry` Timer outer catch â†’ log + no-op (relies on layer a), (c) stream.error listener re-kicks requestReconnect as the fallback if layer b also dies.
- **Files modified:** `lib/features/monitoring/services/reconnect_supervisor.dart`
- **Commit:** `2942ecd`
