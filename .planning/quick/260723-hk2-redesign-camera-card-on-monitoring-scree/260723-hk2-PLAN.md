<objective>
Redesign the `CameraAudioCard` layout so every connection state (live, connecting, reconnecting, error), plus muted / video-on-off / quality / volume, is legible and uncrowded on a ~340-411dp phone. This is a UI-only refactor of a single widget — no player, reconnect, or provider logic changes.

Root causes of the reported problems:
1. The single header row packs status dot + name + status label + 3 icon buttons; on narrow phones the status label truncates to "Re…" and the spinner crowds it.
2. The reconnecting state wraps the header in an amber-tinted `Container` while the live state has a plain header — so the two cards look like different components (structural inconsistency).

Fix strategy (hybrid A+B — user-approved from rendered mockups):
- Split the card into clear zones: a Header zone (identity + action buttons only) and, for healthy states (playing/connecting), a dedicated full-width Status line below it. Moving the status off the header removes all contention and eliminates truncation.
- For problem states (reconnecting/error), render an edge-to-edge tinted STATUS BANNER at the very top of the card (above the header, spanning the full card width, rounded top corners via card clipping). Amber tint + spinner + label for reconnecting; error-red tint + wrapped error text for error. Both problem states use the SAME banner widget in the SAME slot, so cards stay structurally consistent while problems are scannable by color from a distance (tablet grid / 3am glance).
- Healthy cards stay compact (no banner); the banner only appears when something needs attention.

Purpose: A parent glancing at the nightstand phone at 3am must instantly read each camera's state without truncation or a jarring "why does this card look different" moment.
Output: Restructured `camera_audio_card.dart` + updated/extended widget tests.

## Honoring the approved Phase 4 UI-SPEC (READ BEFORE CODING)
This card's states were specified in `.planning/phases/04-reliability-overnight-monitoring/04-UI-SPEC.md` (D-09/D-10/D-11). Two constraints are NON-NEGOTIABLE and the redesign is built to preserve them:

- **D-11 (hard):** The reconnecting card shows status ONLY — NO attempt count ("attempt 3"), NO countdown ("retry in 8s"), NO error message. The task-brief phrase "reconnecting with attempt counts" is superseded by D-11 — attempt counts live on the summary screen, never on the card. Do NOT add them.
- **D-10 (accent contract):** the amber accent (`colorScheme.tertiary` / `tertiaryContainer`) is reserved exclusively for the reconnecting state. Reconnecting MUST still stand out in amber.

Deliberate, documented deviation the user's redesign authorizes: the amber accent moves from a header-wrapping *tint box* (UI-SPEC §Reserved-accent item 1) to an edge-to-edge status banner at the top of the card (tint + spinner + text). The accent color stays reserved for reconnecting only — only its *placement* changes (dedicated banner slot instead of an ad-hoc header wrapper). Error uses the same banner slot in error-red, which is what makes problem cards structurally consistent. Note this in the SUMMARY as a conscious UI-SPEC update.
</objective>
