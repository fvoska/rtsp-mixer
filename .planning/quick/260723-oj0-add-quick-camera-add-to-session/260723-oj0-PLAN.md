<objective>
Add a quick "add camera to the current session" affordance that mirrors the existing quick-remove UX. While monitoring is live, the user can open a picker of cameras that exist but aren't currently in the mix, tap one, and have its audio stream start with sensible defaults — without interrupting any already-running stream.

Purpose: Symmetry with the existing quick-remove feature (`removeCamera` + per-card Remove). The user explicitly asked for the inverse: "I like the ability to quickly remove cameras from current session. add the ability to quickly add them as well."

Output: A new `addCameraToSession` notifier method (single-sourcing the per-camera connect logic already used by `startMonitoring`) and an add-camera affordance + picker in the live monitoring view.
</objective>

  <behavior>
    - Given a live session (players non-empty) and a camera id present in cameraNotifier but NOT in MonitoringState.cameras, addCameraToSession connects that one camera and appends it to MonitoringState.cameras.
    - Adding leaves every pre-existing entry in _players untouched — no existing player is stopped, disposed, or reopened (the observable contract: other cameras' connectionStatus/activeStreamUrl are unchanged across the add).
    - No-op guards: camera already in the mix; camera id not found in cameraNotifier; session not running (players empty). Each guard logs via appLog and returns without mutating state.
    - The added camera restores its saved volume/mute from mix_state and is added to the selected set (persisted) so it survives a restart.
    - A pure, testable helper returns the set of "addable" cameras: cameras in the camera list whose id is not already a cameraId in the current mix. This is what the picker renders and is unit-testable without native libs.
  </behavior>
  <action>
Add `addCameraToSession(String cameraId, {bool videoPreview = false})` to AudioPlayerNotifier, mirroring the existing `removeCamera` (single-camera lifecycle that does NOT disturb the other players).

First refactor to single-source the connect logic (do NOT duplicate the candidate/remote-URL/open block — divergence there would be a reliability bug per CLAUDE.md). Extract the body of the `for (final camera in selectedCameras)` loop inside `startMonitoring` (the block that builds localUrls/remoteQualities/overrideQualities, constructs the CameraAudioState, creates the Player, applies tuning, creates the VideoController, registers listeners, runs `_openFirstCandidate`, sets `vid=no` when not previewing, and records the streamStarted health event / error handoff) into a private helper `Future<CameraAudioState> _connectCamera(ProtectCamera camera, {required bool videoPreview})`. Have startMonitoring's loop call `cameraStates.add(await _connectCamera(camera, videoPreview: videoPreview))`. Keep startMonitoring's outer responsibilities (state=AsyncLoading, beginSession, health clear/monitoringStarted, saved-mix restore over the full list, foreground notification, level polling, connectivity listener) exactly where they are — only the per-camera body moves.

Then implement addCameraToSession:
- Run the whole method through `_runLifecycle` so it can never interleave with a stop/settings-driven restart.
- Read current state via `state.value`; return if null.
- Guard: return (with appLog) if `_players.isEmpty` (no live session), if a camera with this cameraId already exists in `current.cameras`, or if the ProtectCamera can't be resolved from `ref.read(cameraNotifierProvider).value?.cameras`.
- Optimistically append the camera to MonitoringState in `connecting` status is NOT required; instead call `_connectCamera(camera, videoPreview: videoPreview)` and append the returned CameraAudioState via `state = AsyncData(current.copyWith(cameras: [...current.cameras, camState]))`. Do NOT set `state = const AsyncLoading()` and do NOT call stopMonitoring — that would blank the grid and drop the other players. Read `state.value` fresh after the await before appending (the await yields).
- Restore saved volume/mute for just this camera using the same `_loadMixState()` logic startMonitoring uses (look up `savedMix[cameraId]`, apply volume/mute to the new player and copyWith into the appended camState).
- Persist selection: `ref.read(cameraNotifierProvider.notifier).toggleCamera(cameraId)` only if it isn't already selected (toggle would otherwise deselect it) — check `current.selectedIds` from the camera provider's state before toggling.
- Record a `HealthEventType.streamStarted` health event (cameraId + name, detail 'added to mix'), and refresh the foreground notification text the same way startMonitoring does. Wrap health/notification/persistence in try/catch that logs and continues (defensive-error rule).
- The already-running `_levelPollTimer` iterates `state.value.cameras`, so the new camera is picked up automatically — do not restart polling.

Add the pure helper: a top-level function `List<ProtectCamera> addableCameras(List<ProtectCamera> all, List<CameraAudioState> inSession)` returning `all.where((c) => !inSession.any((s) => s.cameraId == c.id)).toList()`. Use it from the UI in Task 2 and cover it with a unit test here.

Every new failure path must be caught+logged and must never propagate in a way that could tear down a running stream (CLAUDE.md: no exception may kill a running audio stream). A failed open for the added camera should leave that one camera in `error` with supervisor/alert handoff (reuse the existing catch block behavior that _connectCamera already carries) while the others keep playing.
  </action>
